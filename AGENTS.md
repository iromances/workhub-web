# AGENTS.md

## 1. 目的

本文件为代码智能体提供在本仓库中工作的最小必要上下文。

- 把本文件视为项目地图，不要把它当成百科全书。
- 更细的业务规则、页面规划、前后端契约和实施路线，以 `docs/` 下文档为准。
- 如果本文件与用户当前指令冲突，以用户当前指令为准。

## 2. 先读哪些文档

开始任何中等及以上规模修改前，先阅读并对照以下文档：

1. `README.md`
2. `docs/project-charter.md`
3. `docs/frontend-architecture.md`
4. `docs/implementation-roadmap.md`

使用原则：

- `README.md` 提供仓库入口、当前已落地页面和启动方式。
- `project-charter.md` 是业务范围、防飘逸约束和生命周期基线的最高优先级文档。
- `frontend-architecture.md` 规定页面地图、目录分层、状态管理和 API 契约原则。
- `implementation-roadmap.md` 规定当前阶段目标、里程碑和验收基线。

## 3. 项目概览

这是 `workhub-web` 前端仓库，基于 Vue 3 + TypeScript + Vite + Pinia + Vue Router + Element Plus + Axios，用于承载：

- 登录与后台工作台
- 项目、迭代、版本、工作项管理页面
- 待整理箱、人工录入、AI 整理确认界面
- 工作项状态流转、跟踪记录、查询和看板

当前代码仍处于“最小可运行骨架向业务页面补齐”的阶段。做功能判断时，优先看代码和 `docs/implementation-roadmap.md`，不要默认某能力已经完整落地。

对应后端项目：

- 后端仓库是同级目录下的 `../workhub-server`

## 4. 不可违背的业务约束

修改前必须确认没有越界：

- 所有外部来源数据先进入待整理箱，再由人工确认生成正式工作项。
- AI 只能生成整理草稿，不能绕过人工确认直接写正式数据。
- 每个工作项必须归属一个项目，运维事项也不例外。
- 正式工作项从 `待澄清` 开始流转，`待整理` 只用于收件箱和草稿。
- 状态变更、指派、转版本、转迭代等动作应使用明确动作入口或独立接口，不要把状态字段当普通表单字段直接保存。
- 前端不直接连接第三方服务，统一经由后端 API。
- 未更新文档前，不新增工作项类型、不改主状态、不引入审批流/权限流/组织流，不引入新的基础设施。

如果需求明显超出上述边界，先更新文档，再改代码。

## 5. 仓库结构

重点目录：

- `src/api/`：接口封装
- `src/types/`：请求和响应类型、领域对象类型
- `src/router/`：路由与菜单结构
- `src/stores/`：登录态、用户信息、全局上下文
- `src/layouts/`：主布局
- `src/views/`：按业务域拆分的页面
- `src/components/`：页面复用组件
- `src/composables/`：可复用组合逻辑
- `docs/`：业务、架构、路线图等真实规则来源

约定：

- 请求参数和响应类型统一定义在 `types` 与 `api` 层。
- 复杂业务规则不要散落在模板表达式中。
- 页面局部表单状态优先保留在页面内部，不随意塞进全局 store。
- 页面、路由、类型、接口按业务域扩展，不要把新增实现零散堆在单一目录。

## 6. 修改代码时的工作方式

1. 先确认变更属于当前路线图阶段范围。
2. 先找已有文档和现有实现，再决定是否新增目录或抽象。
3. 小改动尽量复用现有模式；中大型改动优先补齐 `api/types/views/router` 的对应关系。
4. 改业务页面时，同时检查：
   - 是否需要新增 `src/api/*`
   - 是否需要新增 `src/types/*`
   - 是否需要补路由、菜单、详情入口
   - 是否需要同步更新文档
5. 如发现文档与实现不一致：
   - 若代码明显只是骨架，按文档为准推进
   - 若准备改变既有规则，先更新文档

## 7. 代码风格与实现偏好

- 使用 TypeScript 和 Vue 单文件组件，优先 `script setup`。
- 统一沿用 `@/` 别名导入方式。
- 新增页面优先遵循后台型界面风格：高密度、清晰筛选、明确动作区。
- 不为演示方便发明后端不存在的长期字段；临时占位数据必须易于替换。
- 复杂表单和详情面板优先显式字段配置，不写超长模板。
- 高风险操作应具备明确确认交互。

## 8. 常用命令

- 安装依赖：`npm install`
- 本地开发：`npm run dev`
- 生产构建：`npm run build`

当前仓库未看到成体系自动化测试。提交较大改动前，至少保证：

- `npm run build` 通过
- 受影响页面完成基本手工验证

## 9. 文档维护要求

当你做以下变更时，应同时更新对应文档：

- 新增或移除核心页面
- 改变业务范围、对象模型或生命周期
- 引入新的全局交互规则
- 调整实施阶段、验收基线或开发顺序

优先保持 `AGENTS.md` 简短稳定；详细说明写入 `docs/`，再从这里链接过去。

## 10. 交付预期

完成任务时，尽量说明：

- 改了哪些文件
- 影响了哪个业务域
- 是否更新了文档
- 跑了哪些验证命令
- 还有哪些未完成项或依赖前后端联调

如果以后仓库出现多个相对独立的子模块，可在子目录补充更近一级的 `AGENTS.md`，就近覆盖本文件中不适用的部分。
