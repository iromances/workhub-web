# WorkHub 前端架构说明

## 1. 技术栈

- Vue 3
- TypeScript
- Vite
- Vue Router
- Pinia
- Element Plus
- Axios
- UnoCSS 或 Tailwind 二选一

当前建议：

- 第一阶段优先 `Element Plus + UnoCSS`。
- 原因是后台表单、表格、抽屉、对话框、树筛选、日期区间等场景占比高，Element Plus 更省建设成本。

## 2. 设计目标

1. 面向后台业务，强调密度、筛选、效率和可追踪性。
2. 页面状态简单清晰，避免把业务规则写散在组件里。
3. 所有状态流转必须通过明确的交互入口完成。
4. 所有外部来源数据都先进入待整理视图，再由用户确认。

## 5. 前端状态分层

全局状态：

- 当前登录用户
- token 和登录态
- 字典与通用枚举
- 全局筛选上下文

页面状态：

- 查询条件
- 分页
- 当前 tab
- 抽屉显隐
- 表单草稿

约束：

- 不把表单临时状态塞进全局 store。
- 不把复杂业务规则散落在组件模板中。
- API 请求参数和响应类型统一定义在 `types` 与 `api` 层。

## 6. 交互原则

1. 工作项详情页是核心工作面板，不把关键动作分散到多个页面。
2. 状态流转使用固定动作区，不允许自由修改状态字段后直接保存。
3. 待整理箱要同时展示原始内容、AI 建议、人工确认结果。
4. 高风险操作必须二次确认，例如拒绝、挂起、关闭、恢复暂停需求、删除附件。
5. 主布局导航切换应保留已访问页面的本地查询条件、分页、当前结果和抽屉状态，避免用户在跨菜单核对时丢失上下文。
6. 需求暂停必须通过显式入口录入暂停原因和暂停日期；恢复入口只能调用后端恢复到暂停前状态，页面需能在详情修改历史中查询暂停和恢复记录。

## 7. API 契约原则

1. 列表接口统一返回分页结构。
2. 枚举值传 code，页面展示用 label。
3. 状态变更、指派、转版本、转迭代等动作优先单独接口，不混在通用保存接口里。
4. 所有上传、下载、预览统一封装。

## 8. 质量要求

1. 路由按业务域划分，禁止按页面零散堆积。
2. 复杂表单使用 schema 或显式字段配置，不写超长模板。
3. 对筛选、看板、详情页建立基础组件复用层。
4. 前端页面开发前先对照 `project-charter.md` 检查是否超范围。
