# WorkHub 项目宪章

## 1. 项目定位

WorkHub 是一个面向内部团队的项目与工作项管理系统。第一阶段重点解决企业微信等渠道来的需求、缺陷、运维事项、临时任务无法集中登记、缺少统一跟踪、状态不透明、历史留痕不足的问题。

## 2. 业务目标

1. 建立统一收口。
来自企业微信回调、人工录入、粘贴整理的事项都进入统一待整理箱。

2. 建立统一工作项模型。
需求、缺陷、运维、任务采用统一字段体系和流转规则，避免多个台账并存。

3. 建立全生命周期跟踪。
从登记、澄清、评估、排期、开发、测试、发布到完成，全程留痕。

4. 提高整理效率。
对企微原始消息和人工粘贴文本提供 AI 结构化整理能力，但必须人工确认后入正式工作项。

5. 提高项目透明度。
通过项目、迭代、版本、工作项列表与看板，实现进度可见、责任人明确、历史可追。

## 3. 第一阶段范围

范围内：

- 本地账号密码登录
- 项目管理
- 迭代管理
- 版本管理
- 工作项管理
- 跟踪记录
- 状态流转日志
- 附件关联
- 企业微信入站收件箱
- 企业微信群机器人出站通知
- AI 整理草稿
- 列表、筛选、详情、看板

范围外：

- 企业微信单点登录
- 复杂权限模型和组织架构同步
- 自动审批流引擎
- 自动创建正式需求且不经人工确认
- 多租户
- 工时、成本核算

## 4. 统一对象模型

核心对象：

- 项目
- 迭代
- 版本
- 工作项
- 工作项跟踪记录
- 工作项状态流转日志
- 企微收件箱记录
- 附件
- 用户

工作项类型：

- 需求
- 缺陷
- 运维
- 任务

规则：

- 每个工作项必须归属一个项目。
- 运维事项也必须归属项目，默认通过“公共运维项目”或具体业务项目承载。
- 每个工作项至少包含创建人、负责人、跟进人。
- AI 只能生成草稿，不能跳过人工确认直接入正式台账。

## 5. 生命周期基线

标准状态：

`待整理 -> 待澄清 -> 待评估 -> 待排期 -> 开发中 -> 测试中 -> 待发布 -> 已完成`

补充终态：

- 已拒绝
- 已挂起

状态使用原则：

- `待整理` 只用于收件箱和待转化草稿，不用于正式已确认工作项。
- 正式工作项从 `待澄清` 开始流转。
- 任何状态变更都必须生成流转日志。
- 关闭类状态必须填写结论说明。

## 6. 成功标准

1. 任一来自企微或人工录入的事项都能在系统中找到源记录。
2. 任一正式工作项都能查到负责人、当前状态、所属项目、最近跟进记录。
3. 任一状态变更都能回溯操作者、时间和原因。
4. 项目维度可以看到迭代、版本、工作项分布和当前阻塞项。
5. AI 整理结果能减少人工整理时间，但不会直接污染正式数据。

## 7. 开发原则

1. 先做稳定台账，再做自动化增强。
2. 先确保留痕与追踪，再做智能推荐。
3. 前后端契约优先，避免页面先跑、接口后补。
4. 文档先行，需求扩展先更新项目宪章和实施路线。
5. 能复用统一对象模型时，不为单一场景单独造表。

## 8. 防飘逸约束

以下事项必须先更新文档再开发：

- 引入新的工作项类型
- 改变生命周期主状态
- 让 AI 直接写正式数据
- 新增审批流、权限流、组织流
- 引入新的基础设施，例如消息队列、搜索引擎、对象存储
